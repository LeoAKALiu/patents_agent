# v0.5 Guided Patent Flow 设计

## 目的

当前 `patentAgent` 的能力已经覆盖语料、护城河、前置材料、多 Agent 会审、撰写、提交成熟度、权利要求防线、初稿完善和导出，但这些能力以 12 个并列工作台暴露给用户。这个结构适合研发调试，不适合助手或普通用户从“一段想法”开始完成专利初稿。

v0.5 的目标是把产品主界面改成 **聊天式想法入口 + 五步向导式主流程**。用户或助手只需要输入一段精炼想法，系统引导补充材料和发明点，自动串联现有引擎，最后导出正式提交稿、内部策略稿和完善报告。

本设计不降低专利质量控制要求。`可行未验证`、提交成熟度、权利要求防线和初稿完善仍然保留，只是从“用户必须知道每个模块在哪里”改成“系统在正确步骤自动运行并解释结果”。

## 已确认产品策略

采用 `A+C`：

- `C`：第一屏采用聊天式/助理式输入，让用户用一段话描述想法。
- `A`：后续采用五步向导，而不是让用户在多菜单工作台里自行寻找流程。

默认策略：

- 系统默认一键跑到可导出状态。
- 在两个关键节点暂停确认：
  1. 发明点与护城河确认。
  2. 正式导出前的风险报告确认。
- 高风险仍沿用“警告但允许导出”，不硬阻断导出。

## 目标用户

### 用户助手

助手通常从用户处获得一段精炼想法，需要快速生成可交给用户或代理师审阅的材料。助手不应该理解每个内部模块的区别，只需要知道下一步做什么、哪里需要用户确认、最终导出什么。

### 普通用户

普通用户不理解“前置材料”“Claim Defense Worksheet”“Draft Completion Harness”等术语。界面应使用自然语言表达：输入想法、确认发明点、生成初稿、检查风险、导出文件。

### 高频专利作者

高频用户需要保留专家工具：可以查看护城河地图、权利要求防线、初稿完善矩阵、提交成熟度报告和历史运行记录。但这些不应成为默认路径。

## 信息架构

### 主导航

左侧 12 个并列菜单合并为 3 个一级入口：

1. `专利生成`
   - 默认入口。
   - 包含五步向导。
2. `项目`
   - 项目列表、历史文件、运行记录。
   - 面向复用、继续旧案、查看导出。
3. `专家工具`
   - 语料库建设、知识库检索、护城河地图、前置材料、多 Agent 会审、提交成熟度、权利要求防线、初稿完善、审查修改。
   - 默认折叠，只给高级用户深入查看。

### 第一屏

第一屏直接展示专利助理输入区：

- 大文本框：`把你的想法写成一段话`
- 可选上传：技术交底、聊天记录、论文、方案文档、截图说明
- 目标模式：
  - `授权稳健`
  - `保护范围优先`
  - `快速初稿`
  - `专利护城河`
- 证据策略：
  - 默认允许 `可行未验证` 进入护城河，但正式稿必须标注为可选实施方式或变形例。
- 主按钮：`开始生成专利`

## 五步向导

### 1. 想法与材料

用户输入一段想法，系统创建或更新项目。上传材料后自动解析。页面只显示：

- 想法摘要
- 已上传材料
- 缺失信息提示
- 下一步按钮：`提炼发明点`

内部映射：

- `CreateProjectView`
- `ProjectMaterial` upload
- 原 `前置材料` 中的材料上传能力

### 2. 发明点与护城河

系统自动运行前置材料生成，产出候选发明点、推荐主线、证据状态和支撑缺口。这里暂停给用户确认。

页面显示：

- 推荐主发明点
- 可选从属/分案点
- 每个点的证据状态：`已验证`、`可行未验证`、`需实验`
- 主要风险：现有技术接近、支撑不足、未验证效果
- 操作：`确认主线并继续`、`调整发明点`

内部映射：

- `DisclosureGenerator`
- `PatentPointCandidate`
- `MoatView`
- `PriorArtHit`

### 3. 自动生成初稿

系统根据确认的发明点自动串联：

- 前置交底书
- 可用的多 Agent 会审策略
- RAG 片段
- 专利初稿生成

页面显示：

- 正在执行的任务
- 已完成的阶段
- 当前初稿预览
- 下一步按钮：`运行质量检查`

内部映射：

- `DisclosureRun`
- `DeliberationRun`
- `PatentDraftGenerator`
- `DraftPackage`

### 4. 质量检查与补强

系统自动运行：

- 提交成熟度
- 权利要求防线
- 初稿完善
- 审查意见

页面聚合显示，不要求用户理解每个模块名：

- 总体状态：`可导出 / 建议补强 / 高风险但可导出`
- 三个评分：授权稳定性、保护范围、提交成熟度
- 必看问题：最多 5 条
- 建议补强：可接受或跳过
- 下一步按钮：`进入导出确认`

内部映射：

- `assess_filing_readiness`
- `generate_claim_defense_worksheet`
- `run_draft_completion`
- `PatentDraftGenerator.review`

### 5. 导出

正式导出前第二次暂停确认。页面显示：

- 正式提交稿是否干净
- 是否存在高风险
- 哪些内容只进入内部策略稿
- 可导出文件列表

导出文件：

- `正式提交稿 DOCX`
- `正式提交稿 MD`
- `内部策略稿 MD`
- `提交成熟度报告 MD`
- `初稿完善报告 MD`

内部映射：

- `official-export.docx`
- `official-export.md`
- `export.md`
- `filing-readiness/{id}/export.md`
- `completion-runs/{id}/report.md`

## 专家工具保留方式

专家工具不删除现有能力，只改变入口层级：

- 语料库建设和知识库：归入 `专家工具 / 知识库`
- 护城河地图：归入 `专家工具 / 发明点`
- 前置材料：归入 `专家工具 / 交底`
- 多 Agent 会审：归入 `专家工具 / 策略`
- 提交成熟度、权利要求防线、初稿完善、审查修改：归入 `专家工具 / 质检`
- 导出：主流程保留，专家工具也可直接进入

这样高频用户仍可精细控制，但普通用户默认不需要触碰这些内部模块。

## 状态与进度

主流程需要一个统一的项目流程状态，而不是让 UI 分散读取多个模块。

建议前端先实现派生状态，不急于新增数据库表：

- `idea_ready`
- `materials_ready`
- `invention_points_ready`
- `draft_ready`
- `quality_checked`
- `export_ready`

每个状态由现有数据推导：

- 是否有项目 draft
- 是否有 processed materials
- 是否有 completed disclosure run 或 selected patent point
- 是否有 package
- 是否有最新 filing readiness / worksheet / completion run
- 是否可导出 official package

后续可增加 `workflow_runs` 表，但 v0.5 首版不需要。

## 错误处理

主流程必须把内部错误翻译成用户能理解的动作：

- LLM 连接失败：提示“模型服务连接失败，请检查代理/API key”，并允许重试。
- 前置材料检索失败：提示“公开检索不可用，本轮将基于本地材料生成”，不阻断。
- 多 Agent 会审失败：提示“会审失败，已使用默认策略继续”，不阻断。
- 高风险质检：提示“高风险但允许导出”，并展示风险报告。
- 长任务运行：显示当前阶段和预计下一步，避免只显示 `queued/running`。

## 前端改动范围

新增：

- `GuidedPatentFlowView`
- `IdeaIntakePanel`
- `WorkflowStepper`
- `InventionPointConfirmation`
- `QualitySummary`
- `ExportConfirmation`

调整：

- `workspaceTabs` 改为主入口：`专利生成`、`项目`、`专家工具`
- 现有视图作为专家工具子视图复用
- 顶部项目选择保留，但在主流程中弱化为“当前项目”
- 页面文案从内部模块名改为用户动作语言

## 后端改动范围

首版优先复用现有 API，不强行改后端：

- 创建项目：`POST /api/projects`
- 上传材料：`POST /api/projects/{id}/materials`
- 前置材料：`POST /api/projects/{id}/disclosures`
- 撰写：`POST /api/projects/{id}/generate`
- 提交成熟度：`POST /api/projects/{id}/filing-readiness`
- 权利要求防线：`POST /api/projects/{id}/claim-defense-worksheets`
- 初稿完善：`POST /api/projects/{id}/completion-runs`
- 导出：现有 export endpoints

可以新增一个轻量编排 API，但不是首版必需：

- `POST /api/projects/{id}/guided-runs`
- `GET /api/projects/{id}/guided-status`

首版建议先在前端串联现有 API，降低风险。

## 测试策略

前端：

- 主入口只显示 3 个一级入口。
- 默认进入 `专利生成`。
- 输入一段想法后能创建项目并进入步骤 2。
- 有 completed disclosure run 时步骤 2 显示发明点确认。
- 有 package 后步骤 4 可运行质检。
- 高风险状态仍可进入导出确认。
- 专家工具仍能进入所有旧模块。

后端：

- 现有 API 行为不变。
- 若新增 guided status API，测试状态推导覆盖空项目、已有材料、已有初稿、已有质检和可导出状态。

端到端手工验收：

1. 用一句话创建新专利项目。
2. 上传一份补充材料。
3. 生成并确认发明点。
4. 生成初稿。
5. 运行质量检查。
6. 在高风险提示下仍能导出正式稿和内部报告。

## 非目标

v0.5 不做：

- 自动替代代理师判断。
- 自动提交专利。
- 删除专家工具。
- 完整重写后端编排。
- 重新设计专利质量评分算法。
- 长时间无人值守多轮优化。

本版本只解决一个核心问题：让用户从“一段想法”清楚地走到“可导出专利文件”。
