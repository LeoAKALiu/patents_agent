# PatentAgent v1.1.0 quality gates

PR7 adds a deterministic golden E2E gate for the v1.1.0 release train. The gate is designed for local CI and GitHub Actions without live provider credentials.

## Single command

Run the full deterministic suite from the repository root:

```bash
scripts/v1_smoke.sh
```

The command runs backend pytest, the v1.1 golden API quality gate, frontend tests/build, desktop build, and the desktop smoke when feasible. The API quality gate runs two deterministic rounds by default to catch loop repeatability drift. The v1.1 gate report is written to:

```text
.artifacts/v1.1.0-quality/v1_1_quality_report.json
.artifacts/v1.1.0-quality/v1_1_quality_report.md
```

Override the report destination when needed:

```bash
PATENTAGENT_V1_1_REPORT_DIR=/tmp/patentagent-v1.1-quality scripts/v1_smoke.sh
```

Override the repeat count when diagnosing a local failure:

```bash
PATENTAGENT_V1_1_REPEAT_COUNT=3 scripts/v1_smoke.sh
```

## Golden coverage

The deterministic API gate covers checked-in samples for:

- software invention
- sensing and inspection invention
- mechanical/device utility model
- algorithmic invention
- external draft intake

Each workflow verifies:

- prior-art evidence count
- grantability report and claim chart presence
- quality trend output for authorization stability, support strength, prior-art distinction, official hygiene, and overall score
- drafting-quality metrics for evidence binding rate, core feature support rate, unsupported core feature count, unverified effect leakage, dependent fallback depth, embodiment density, and patch delta
- official export blocked before compile
- official export blocked before post-draft review
- official export unlocked only after post-draft review
- official export hygiene with internal markers excluded

## Loop Engineering Pipeline Tests

The pipeline test standard is now expressed as a loop-engineering gate: the
objective is `final patent draft quality`, and the standard is
`full-process stability and reliability`.

The deterministic API gate records three loop groups in
`loop_engineering.gate_groups`:

- `stability`: quality trend fields are present and official export hygiene
  stays clean.
- `reliability`: official export fails closed before compile, stays locked
  before post-draft review, and unlocks only after a passing post-draft review.
- `final_draft_quality`: prior-art evidence, grantability claim chart,
  evidence binding rate, core feature support rate, and unverified-effect
  leakage are all checked before the final official draft is considered ready.

For repeatability, run the deterministic API gate with at least two rounds:

```bash
python3 scripts/v1_api_smoke.py \
  --repeat-count 2 \
  --report-dir .artifacts/v1.1.0-quality-loop
```

The repeatability check compares stable quality signatures while ignoring
per-run ids such as project, official compile, and post-draft review ids. Any
drift in the official text hash, quality trend, drafting-quality metrics,
grantability summary, evidence count, or gate outcomes is classified as a
`code` failure. The artifact-level official package hash is still reported for
traceability, but it is not used for cross-run repeatability because it includes
run-bound source hash material.

## Drafting-Quality Gates

These gates track whether the draft is becoming more evidence-backed and easier to finish:

- `evidence_binding_rate >= 0.1`
- `core_feature_support_rate >= 0.4`
- `unverified_effect_leak_count == 0`

The floors are intentionally conservative so the default deterministic suite detects regressions without requiring live-provider behavior or project-specific evidence volume.

## Export Hygiene Gates

Export gates remain separate from drafting-quality metrics:

- official export must be blocked before official compile
- official export must be blocked before post-draft review
- post-draft review must unlock export only for the current official package
- official markdown must exclude internal markers such as generation logs, prompts, Mermaid fences, support-gap labels, and sidecar reports

## Failure classification

The JSON report records failed workflows with a classification:

- `code`: deterministic product regression or gate assertion failure
- `environment`: missing local runtime, sample, package, or desktop display dependency
- `unavailable_provider`: live provider or credential unavailable

The default v1.1 gate uses `TestClient` and `V1SmokeLLM`; it does not read `.env`, does not require `DEEPSEEK_API_KEY`, and does not call live provider APIs.

## Live provider boundary

Live-provider smoke remains opt-in through the existing provider-specific test environment variables. Do not enable live-provider secrets in default CI. Live runs should be used as diagnostic evidence only and must not weaken deterministic release gates.
