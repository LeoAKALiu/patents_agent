# OMP UI Refactor Design Contract

This contract is the design source for OMP implementation work on the PatentAgent production UI. It updates the older UI redesign contract for the current branch state.

This document is a requirement, not implementation evidence. Production proof must come from changed files under `frontend/src/`, current-source dev screenshots, targeted tests, and, for desktop packaging, evidence from the exact packaged artifact.

## Product Direction

PatentAgent should feel like a quiet professional workbench for repeated patent drafting and repair work. The UI should favor dense but readable task surfaces over marketing composition, oversized hero sections, decorative cards, or illustration-led pages.

The current implementation already has the main shell, top-level navigation, workbench route, document repair workspace, export workspace, knowledge workspace, expert tools workspace, and embedded annotated repair editor. The next OMP phase should harden the implementation rather than restart it.

## Locked IA

The left sidebar remains destination-only:

```text
工作台
项目
文稿与修复
知识库
专家工具
导出
设置
```

The nine guided steps remain a project progress model inside the workbench. They must not return as global navigation.

## Surface Contracts

### Shell

- Sidebar and topbar may use restrained glass treatment.
- Shell chrome is compact, stable, and navigation-first.
- The topbar chip order remains project selector, export status, run status, backend status, refresh.
- Backend diagnostics live behind a dialog or drawer trigger, not in expanded sidebar content.
- Mobile navigation must expose all seven destinations without text clipping.

### Workbench

- The workbench is the current project command center.
- With no selected project, it shows start paths and does not collapse into a blank or alarm-heavy screen.
- With a selected project, it emphasizes one next action and lets secondary routes sit behind compact controls.
- The workbench must not render raw JSON keys, run IDs, full hashes, or log content by default.
- Information density should be useful, not decorative. Remove duplicated right-rail summaries unless they answer a distinct operational question.

### Document And Repair

- `文稿与修复` owns document status, internal draft editing, issues, annotated repair, and version chain.
- The tabs remain: `总览`, `编辑`, `问题`, `标注修复`, `版本`.
- Gate labels must use the approved vocabulary:
  - `可编辑`
  - `当前有效`
  - `已失效`
  - `需要修复`
  - `待重新验证`
  - `导出锁定`
  - `可导出`
  - `运行中`
  - `等待生成`
  - `运行失败`
- Official submission material and internal repair or strategy material must look visibly distinct.
- Dense document, issue, risk, and editor panes should use solid readable surfaces, not heavy glass.

### Annotated Repair

The annotated repair editor must satisfy the real data-flow gate, not only DOM evidence:

- A selected project has a post-draft review run eligible to open repair.
- `GET /api/projects/{project_id}/post-draft-reviews/{run_id}/repair-session` returns non-empty `issues` and non-empty draft `sections`.
- The editor shows a left issue rail, middle draft sections, and right issue detail panel.
- Selecting an issue updates the highlighted draft section and inspector.
- Long issue lists scroll inside their pane and do not push the draft editor off screen.
- Applying a patch marks the issue as pending revalidation in the rail and inspector.

### Quality Panel

- The guided quality panel should summarize only workflow-critical quality state.
- Full reports remain in expert tools.
- The three entry cards are `提交成熟度`, `权利要求防线`, and `初稿完善`.
- `一键接受补强` is disabled when there are no proposed patches and calls `onAcceptAllPatches` only when a completion run and proposed patches exist.
- Card actions must not stretch or wrap awkwardly on mobile.

### Export

- Export remains a top-level destination.
- Export readiness may be summarized in workbench and document repair, but final file handoff belongs to export.
- The UI should distinguish `正式提交稿`, `内部复核材料`, and trace or risk evidence.

## Visual Contract

- Use the existing React/Tauri design system and token files first.
- Keep the light-first professional palette. Do not introduce a new dominant hue family.
- Use glass only on shell chrome, compact chips, drawers, overlays, and small status surfaces.
- Use solid white or neutral surfaces for editors, document panes, issue lists, risk lists, and tables.
- Card radius should stay at 8px or less unless the local component already requires otherwise.
- Do not nest cards inside cards.
- Do not add decorative gradient orbs, bokeh blobs, SVG hero art, or marketing-style hero sections.
- Use lucide icons in icon buttons and action labels where an icon exists.
- Prefer familiar icon buttons for repeated tool actions and text buttons for clear commands.
- Text must not overflow its container on mobile or desktop.
- Do not scale font size with viewport width.
- Letter spacing should remain 0.

## Interaction Contract

- Every visible tab, button, menu, and diagnostic trigger must be functional.
- Disabled states need a visible reason through nearby copy, title text, or gate explanation.
- Hover and focus states should be visible without shifting layout.
- Keyboard focus must be visible for primary actions, tabs, issue items, and topbar diagnostics.
- Mobile layouts must avoid horizontal overflow and keep action buttons tappable.

## Copy Contract

- Interface copy stays Chinese.
- Prefer task-state copy over implementation terms.
- Do not expose raw keys such as `generation_logs` or `official_safe_patches`.
- Avoid optimistic completion language when an artifact can be stale.
- Use `导出锁定`, `待重新验证`, and `已失效` when gates are not current.

## Verification Contract

Each OMP PR must include:

- Source identity: branch, short SHA, worktree path, dirty status.
- Exact changed files.
- Targeted Vitest command and result.
- `npm run build` or an explicit reason it was not run.
- Screenshot evidence for desktop and mobile when UI changed.
- For annotated repair, API/data-flow evidence from a real repairable review, not only mocked DOM tests.
- Confirmation that no stale `/Volumes/PatentAgent` artifact was used for packaged evidence.
