## Confirmed Issues

### BUG-117-01: SettingsPanel save/load/clear errors still use LLM error helper

- Severity: P2
- Scenario: ordinary backend validation error from `PATCH /api/desktop-config`
- Reproduction: mocked `updateDesktopConfig` rejects with `PATCH /api/desktop-config 返回 422：Base URL must start with http:// or https://.`
- Actual: `SettingsPanel` renders `设置保存失败：请检查配置或稍后重试；如果问题持续存在，请保留诊断详情用于排查。`
- Expected: generic app copy such as `输入未通过校验：Base URL must start with http:// or https://.`
- Stable: yes, reproduced by `frontend/src/SettingsPanel.test.tsx`
- Likely root cause: `frontend/src/SettingsPanel.tsx` imports and calls `userFacingErrorMessage` for load/save/clear, while app/backend errors should use `userFacingAppErrorMessage`.
- Recommended fix: switch SettingsPanel load/save/clear paths to `userFacingAppErrorMessage`; keep `userFacingErrorCopy` for explicit LLM health-check results.
- Recommended regression: keep the new SettingsPanel validation-copy test and add load/clear variants for 404/409/500.

### DOC-117-01: QA pipeline doc still had two BUGS.md-as-existing prompts

- Severity: P3
- Scenario: first use before a defect ledger exists
- Actual before audit fix: later prompts referenced `BUGS.md` directly as though it already existed.
- Expected: refer to the defect ledger created for the current run, with `BUGS.md` as an example.
- Status: fixed in `docs/qa/ai-scenario-testing-pipeline.md` during this audit.
- Recommended regression: static docs check for hard-coded required `BUGS.md` wording unless paired with first-use/create language.

## High-Risk Unconfirmed Areas

- Multi-tab duplicate submissions: backend official compile and post-draft review endpoints are synchronous and do not appear to reject duplicate active requests; single-tab UI busy state may not protect two tabs.
- Material list refresh proof is still too shallow at unit level. `handleUploadMaterial` does call `loadMaterials(projectId)` when any upload succeeds, but an integration/e2e test should prove the refreshed list renders the later valid file after the first selected file fails.
- Upload cancellation/refresh is not fully covered. Browser refresh can drop the toast, while persisted valid files should recover from list API.
- Runtime-control edits during active review/repair need a real UI exploration pass to confirm whether users can mutate provider settings mid-run.
- Large mixed upload batches may stress concurrent `Promise.all` behavior and summary readability.
- Scope breadth: PR #117 includes broad UI state/style changes such as `frontend/src/app/AppRoot.tsx` and about 266 inserted lines in `frontend/src/styles.css`. They may be related to recovery/status UI, but should receive explicit reviewer attention because they are wider than the three known information-level fixes.
