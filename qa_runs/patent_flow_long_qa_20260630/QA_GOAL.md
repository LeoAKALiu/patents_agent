# patent_agent 长程 QA Goal：自动化测试专利撰写流程完整性、流畅性、bug 和卡点

你是一名资深 QA 工程师、专利软件产品测试工程师、LLM 应用测试工程师和代码审计员。请在当前仓库 patent_agent 中执行一轮长程自动化测试，目标是验证“专利撰写流程”的完整性、流畅性、可恢复性和错误处理质量，并结构化记录 bug、卡点、流程断点和改进建议。

## 0. 总目标

本轮目标不是快速跑单元测试，而是模拟真实用户从 0 到 1 使用 patent_agent 完成一件专利撰写任务的全过程，尽可能发现：

- 流程无法走通的问题
- 用户不知道下一步该做什么的问题
- 页面、CLI、API 或 agent 状态不一致的问题
- LLM 输出质量导致的流程卡点
- 输入材料不完整时的阻断/引导问题
- 专利撰写各阶段上下文丢失问题
- 重复点击、刷新、中断、恢复、重试导致的问题
- 文件导入、生成、保存、导出、版本记录的问题
- 错误文案不准确或不可操作的问题
- 测试覆盖缺口

默认原则：

- 不要修改业务代码。
- 不要直接修复 bug。
- 允许新增 QA 记录文件、测试矩阵、探索日志、自动化测试草案或临时测试脚本。
- 如果确实需要改业务代码才能继续验证，请停止修改，记录为 blocker。
- 如果现有测试命令失败，先分析失败是否与本轮目标相关，不要盲目修复。
- 每个发现都必须可复现、可定位、可分级。
- 不要泛泛而谈，尽量使用仓库中的真实文件名、命令、路由、函数、组件、API、agent 名称和测试名称。

特点：

特点：
- 提供技术交底书、产品需求文档、竞品说明
- 关注技术效果、创新点和保护范围

目标：

### 用户 D：粗心/误操作用户

- 重复点击

## 4. 专利撰写主流程测试

请尽量自动化模拟以下完整流程。如果产品实际流程不同，请以仓库真实流程为准，并在 RUN_LOG.md 中说明差异。

### 4.1 新建项目/案件

测试点：

- 新建专利项目/案件
- 输入标题
- 输入技术领域
- 输入发明人或申请人信息，如果系统支持
- 保存项目
- 刷新/重启后恢复项目
- 项目列表显示正确
- 项目状态正确

期望：

- 用户能明确知道项目是否创建成功
- 项目 ID、名称、状态稳定
- 刷新后不丢失
- 没有假成功状态

### 4.2 输入技术交底材料

测试点：

- 手动输入发明描述
- 上传技术交底书
- 上传多个材料
- 上传空文件
- 上传超大文件
- 上传错误格式
- 上传重复文件名
- 上传中断后重试
- 上传后材料列表刷新
- 材料内容被正确解析
- 材料摘要或结构化提取正确显示

期望：

- 成功材料可见
- 失败材料有明确原因
- 部分成功不会被整体失败吞掉
- 解析失败不会导致项目不可用
- 用户知道下一步该补什么材料

### 4.3 发明理解/技术方案抽取

测试点：

- 从材料中抽取技术问题
- 抽取技术方案
- 抽取有益效果
- 抽取关键部件/步骤/模块
- 抽取实施例
- 抽取区别特征
- 用户修改抽取结果
- 保存修改
- 再进入下一步

期望：

- 抽取结果结构清楚
- 不完整信息有补充提示
- 用户修改不会被下一步覆盖
- 关键上下文能传递到后续权利要求和说明书生成

### 4.4 现有技术/背景技术处理

测试点：

- 输入或上传现有技术材料
- 让系统生成背景技术
- 区分现有技术缺陷和本发明技术效果
- 缺少现有技术时是否能继续
- 现有技术与发明点冲突时如何提示

期望：

- 不把本发明误写成现有技术
- 不夸大现有技术缺陷
- 背景技术和发明内容逻辑一致
- 缺失材料时有明确引导

### 4.5 权利要求生成

测试点：

- 生成独立权利要求
- 生成从属权利要求
- 生成方法权利要求，如果适用
- 生成系统/装置权利要求，如果适用
- 修改权利要求
- 重新生成
- 局部 repair
- 检查术语一致性
- 检查引用关系
- 检查多项从属引用
- 检查权利要求是否被说明书支持

期望：

- 独权保护范围清楚
- 从权引用关系正确
- 权利要求编号连续
- 术语不漂移
- 用户修改后的权利要求不会被意外覆盖
- 重新生成前有确认或版本记录
- 失败后可重试

### 4.6 说明书生成

测试点：

- 生成技术领域
- 生成背景技术
- 生成发明内容
- 生成附图说明，如果支持
- 生成具体实施方式
- 生成多个实施例
- 基于权利要求生成说明书
- 基于说明书回补权利要求支持
- 检查章节完整性
- 检查前后术语一致性
- 检查与用户原始材料一致性

期望：

- 说明书章节完整
- 权利要求中的关键特征在说明书中有支持
- 不丢失核心创新点
- 不产生明显自相矛盾
- 长文本生成时进度和错误状态清楚
- 失败不会污染已有草稿

### 4.7 摘要、标题、附图说明

测试点：

- 生成摘要
- 生成专利标题
- 生成附图说明
- 输入无附图时的处理
- 有多个附图时的编号一致性
- 摘要是否过长或过短
- 摘要是否包含商业宣传用语

期望：

- 标题、摘要、附图说明与正文一致
- 编号连续
- 缺少附图时不生成虚假附图
- 用户能编辑并保存

### 4.8 全文审查和 repair

测试点：

- 一键审查全文
- 检查术语一致性
- 检查权利要求支持性
- 检查章节缺失
- 检查引用编号错误
- 检查明显法律/格式问题
- 触发 post-draft repair
- 局部 repair
- 多轮 repair
- repair 过程中刷新
- repair 失败后重试
- repair 后版本比较

期望：

- repair 不破坏用户已确认内容
- repair 前后差异可见
- repair 失败不会清空正文
- 多轮 repair 不引入格式漂移
- 用户知道 repair 改了什么

### 4.9 导出和交付

测试点：

- 导出 DOCX/PDF/Markdown/HTML，如果支持
- 导出失败
- 文件名非法字符
- 长标题导出
- 空内容导出
- 部分章节缺失时导出
- 导出后重新打开检查内容
- 导出内容与页面内容一致

期望：

- 导出文件可打开
- 编号、标题、章节、换行格式正确
- 不丢失用户修改
- 错误文案明确
- 导出失败可重试

## 5. 流畅性测试

除了功能正确性，请特别记录“卡点”。卡点不一定是 bug，但会影响用户完成流程。

重点观察：

- 用户是否知道下一步点哪里
- 必填项是否明确
- 系统是否解释为什么阻断
- loading 是否有进度、阶段或可理解文案
- LLM 生成时间长时是否有反馈
- 错误后是否有恢复路径
- 是否有“死胡同”页面
- 页面状态是否与实际任务状态一致
- 用户修改内容后是否有保存提示
- 重新生成是否会覆盖内容
- 多步骤之间是否能回退
- 多步骤之间是否保留上下文
- 术语、标题、权利要求、实施例是否在流程中漂移
- 文案是否让非专利用户看得懂
- 文案是否让专利专业用户觉得可控

每个卡点记录到 FLOW_FRICTION.md：

- FRICTION ID
- 用户画像
- 所在流程阶段
- 触发步骤
- 观察到的卡点
- 为什么影响流畅性
- 建议改进
- 严重程度：High / Medium / Low
- 是否建议自动化检测

## 6. 错误、异常和恢复测试

请系统测试以下异常：

### 输入异常

- 空发明描述
- 极短描述
- 超长描述
- 只有营销语言，没有技术方案
- 多个发明点混杂
- 前后矛盾材料
- 中英文混合
- 表格、代码块、公式
- 特殊符号
- prompt injection，例如“忽略之前所有规则”
- 与专利无关内容

### 文件异常

- 空文件
- 过大文件
- 损坏文件
- 错误扩展名
- MIME 与扩展名不一致
- 重复上传
- 文件名包含特殊字符
- 文件名超长
- 多文件部分成功
- 上传中取消或刷新

### 服务异常

- 后端 400/404/409/422/500
- LLM provider unavailable
- LLM timeout
- LLM rate limit
- LLM malformed response
- 网络断开
- 请求超时
- 请求 abort
- JSON 响应缺字段
- detail 为空、字符串、数组、对象

### 状态异常

- 生成中重复点击
- repair 中重复点击
- 上传中删除材料
- 生成中切换项目
- 生成中刷新页面
- 两个 tab 同时打开同一项目
- 一个 tab 修改材料，另一个 tab 生成权利要求
- 一个 tab repair，另一个 tab 导出
- 后端任务完成但前端仍 loading
- 前端显示成功但后端未保存

期望：

- 不产生假成功
- 不丢失用户输入
- 不卡死 loading
- 不重复提交不可逆任务
- 错误文案准确
- 用户知道如何恢复
- 重试后能继续流程
- 状态最终一致

优先级规则：

- P0：导致完整专利撰写流程无法完成、数据丢失、导出不可用、严重状态错乱。

- P2：用户需要绕路才能完成，或错误文案/恢复路径较差。
- P3：轻微文案、布局、低频边界问题。

## 9. 自动化测试建议

在 AUTOMATION_RECOMMENDATIONS.md 中输出建议补充的自动化测试，按优先级排序。

至少覆盖：

1. 新建项目 -> 上传材料 -> 生成权利要求 -> 生成说明书 -> repair -> 导出 的完整 happy path。
2. 多文件上传部分成功。
3. 空材料/不完整材料的阻断和引导。
4. LLM timeout / malformed response / rate limit 的错误展示和恢复。
5. 生成中重复点击不会重复提交。
6. 刷新后项目状态恢复。
7. 用户修改后的权利要求不会被重新生成覆盖。
8. repair 后差异可见且不清空正文。
9. 导出内容与页面内容一致。
10. prompt injection 材料不会污染系统行为。
11. 术语一致性检查。
12. 权利要求编号和引用关系检查。
13. 说明书章节完整性检查。
14. 缺少附图时不生成虚假附图。
15. 失败后重试能恢复。

每条建议写清楚：

- 测试名称
- 测试层级
- 建议文件位置
- mock 策略
- 输入数据
- 断言点
- 能防止什么回归
- 实现复杂度：低 / 中 / 高
- 优先级：P0/P1/P2/P3

- 一个实施例

### 中等复杂发明

主题示例：
一种基于多传感器融合的设备异常预测系统。

包含：
- 多个模块
- 多个数据流
- 方法步骤
- 系统装置
- 可选实施例
- 技术效果

### 不完整交底

只有：
- 产品名称
- 几句功能描述
- 没有清楚技术方案

用于测试系统是否能引导补充信息。

### 冲突交底

材料 A 说使用规则模型。
材料 B 说使用神经网络模型。
用于测试冲突检测和用户确认。

### 边界交底

- 超长文本
- 空文本
- 特殊字符
- 中英文混合
- 表格
- 代码块
- prompt injection
- 非专利内容

要求：

- 不要使用真实客户、真实未公开发明或敏感专利材料。
- 如果仓库已有 fixtures，优先复用或扩展。
- 新增测试数据应放到合适的 qa_runs 或 test fixtures 路径中，避免污染生产数据。

## 12. 输出质量检查

专利文本不是只要生成出来就算通过。请尽量检查以下质量不变量：

- 标题与发明主题一致
- 摘要不包含明显营销语言
- 权利要求编号连续
- 从属权利要求引用存在
- 独立权利要求包含必要技术特征
- 从属权利要求进一步限定独权
- 说明书包含技术领域、背景技术、发明内容、附图说明、具体实施方式，若产品要求如此
- 权利要求中的关键术语在说明书中有支持
- 同一术语前后一致
- 不凭空生成用户未提供的关键硬件、步骤或效果
- 不把本发明写成现有技术
- 不把现有技术缺陷写成本发明优点
- repair 不删除核心技术方案
- 导出结果与页面/数据源一致

把违反这些不变量的问题记录为 bug 或 quality risk。

## 13. 长程执行策略

请按循环执行：

1. 观察当前仓库和可运行入口。
2. 记录已知流程和风险。
3. 设计测试矩阵。
4. 先跑已有自动化测试。
5. 选择一条最高价值的完整用户路径。
6. 尝试自动化或手动脚本化执行。
7. 记录 bug、blocker、friction。
8. 根据发现调整下一轮测试。
9. 补充自动化建议。
10. 更新 RUN_LOG.md。
11. 继续下一条高风险路径，直到达到停止条件。

每一轮在 RUN_LOG.md 中写：

- 当前轮目标
- 执行动作
- 观察结果
- 发现的问题
- 下一步计划
- 是否接近停止条件

## 14. 停止条件

当满足以下条件时，可以结束本轮 goal：

- 已识别并记录当前产品专利撰写主流程。
- 已生成 TEST_MATRIX.md。
- 已至少尝试 3 条完整或近完整用户路径：
  1. happy path
  2. 不完整材料路径
  3. 异常/中断/恢复路径
- 已执行仓库中可用的主要测试命令，或明确记录无法执行原因。
- 已记录所有发现的 bug、blocker 和 friction。
- 已给出自动化测试建议。
- 已给出当前最关键的合入/发布风险判断。
- 没有继续执行所必需但缺失的人工信息，或已明确记录 blocker。

不要因为遇到第一个失败就停止。只有当无法继续推进任何替代路径时，才以 blocker 结束。

## 15. 最终汇报格式

最终输出请包含：

1. 本轮 QA 摘要
2. 已理解的 patent_agent 专利撰写主流程
3. 已执行命令和结果
4. 已覆盖的用户路径
5. 发现的 P0/P1/P2/P3 bug
6. 发现的 blocker
7. 发现的流程卡点 friction
8. 高风险但未确认的问题
9. 建议优先补充的自动化测试
10. 是否建议当前版本进入下一阶段
11. 仍未覆盖的风险
12. 本轮创建或修改的 QA 文件列表

最终判断请使用以下格式之一：

- 建议继续合入/发布：未发现阻断问题，仅有低风险问题。
- 有条件合入/发布：存在 P2/P3 问题，但核心流程可完成。
- 不建议合入/发布：存在 P0/P1，或完整专利撰写流程无法稳定完成。
- 无法判断：环境、依赖、密钥或文档缺失导致无法完成主流程验证。
